TSDTech — Product Overview

Plataforma de gateway de pagamentos white-label. Atua como camada de abstração entre um banco (Pinbank) e serviços de domínio, rodando em microsserviços NestJS no GCP.

Core Capabilities

  • Pagamentos PIX — QR Code, devoluções, splits
  • Conta Digital — PF/PJ, saldo, extrato via Pinbank
  • Subcontas e Ledger — saldo duplo (pending + settled), auditoria por ledger entries
  • Gateway de Pagamentos — Pix, boleto, cartão
  • Billing e Report — faturamento e conciliação

Target Use Cases

  • Fintechs que precisam de infra bancária white-label
  • Gateway de pagamentos white-label para fintechs
  • Operações de depósito, saque e split de valores

Value Proposition

  • Ledger de auditoria próprio (NÃO double-entry — modelo PENDING→SETTLED com reversão)
  • Saldo duplo — pendingBalance (pendente) + balance (liquidade) para evitar saldo negativo
  • Encrypted-first — endpoints criptografados por padrão no bus layer
  • DDD Anêmico — separação rígida Controller → Service → DAO/Repository → Entity

TSDTech — Technology Stack

Architecture

Microsserviços com DDD Anêmico no padrão NestJS modular. Bus layer abstrai banco (Pinbank); domain services contêm regras de negócio. Comunicação assíncrona via GCP Pub/Sub.

Core Technologies

  • Language: TypeScript (strict mode)
  • Framework: NestJS
  • Runtime: Node.js 20+
  • Precisão Financeira: Decimal.js (nunca number para valores monetários)

Databases

Bus (pinbank-bus, banking) Firestore BaseDao<T> SerializeUtils.serializer
Banking (ms-banking) Firestore + AES BaseDao<T> (com crypto) SerializeUtils.serializer
Domain (subaccount, report) PostgreSQL BaseRepository<T, F> (Knex) DefaultSerializer
Cache Redis — —

Key Libraries

  • Serialization: typescript-json-serializer (@JsonObject, @JsonProperty)
  • Validation: class-validator + class-transformer
  • API Docs: @nestjs/swagger (@ApiProperty, @ApiResponse)
  • ORM: Knex.js (PostgreSQL), @google-cloud/firestore (Firestore)
  • Queue: @google-cloud/pubsub
  • Encryption: AES (dados bancários em repouso)
  • Testing: Jest

Development Standards

Type Safety

  • TypeScript strict mode
  • noImplicitAny, strictNullChecks
  • Proibido any — usar unknown se necessário

Code Quality

  • PascalCase para classes e arquivos de módulo
  • kebab-case para arquivos de serviço/DAO

Testing

  • Jest com cobertura mínima 80%
  • Testes unitários para Services (lógica de negócio)
  • Testes de integração para fluxos críticos

Development Environment

Required Tools

  • npm/yarn
  • GCP CLI (gcloud) para emuladores
  • Ollama (para tooling local)

Key Technical Decisions

  1. Ledger NÃO é double-entry — modelo próprio com estágios PENDING→SETTLED e reversalOfLedgerEntryId para compensação
  2. Valores em centavos na API — input/output em centavos (integer), conversão para Decimal internamente
  3. Endpoints criptografados por padrão — @PinbankProtected() com encryption: true no bus layer
  4. Múltiplos guards em cadeia — MasterAuth → AgentAuth → C2cAuth → GcpAuth → MsAuth

TSDTech — Project Structure

Organization Philosophy

DDD com camadas isoladas. Cada feature é um bounded context dentro de modules/api/. Separação rígida: Controller (thin) → Service (business logic) → DAO/Repository (data access) → Entity (domain model). NUNCA cruzar camadas.

Directory Patterns

src/
├── index.ts                  # Bootstrap (NestFactory, global pipes/guards/interceptors)
├── main.ts                   # Entry point
├── core/                     # Shared infrastructure
│   ├── base/                 # Base classes (BaseDao, BaseRepository, BaseClient, PubSubBaseService)
│   │   ├── base.dao.ts       # Firestore generic DAO
│   │   ├── base.repository.ts # Knex generic repository
│   │   ├── base.client.ts    # HTTP client base
│   │   └── pubsub/           # GCP Pub/Sub integration
│   ├── constants/            # App-wide constants (port, env vars)
│   ├── decorators/           # Custom decorators (@KnexDBDate, @KnexDBBool, @DecimalJsonSerializer)
│   ├── dtos/                 # Shared DTOs (PaginationDto, PaginatedListDto)
│   ├── entities/             # Base entities (ImmutableEntity, MutableEntity)
│   ├── error/                # HTTPError, ErrorFilter, DatabaseError
│   ├── guards/               # Auth guards (C2cAuth, MsAuth, AgentAuth, GcpAuth, MasterAuth)
│   ├── interceptors/         # Global interceptors (UnixTimestamp, ClassSerializer)
│   ├── interfaces/           # Shared interfaces
│   ├── serializer/           # DefaultSerializer wrapper
│   ├── services/             # Shared services (Redis, cache)
│   └── utils/                # Utilities (Swagger, Firestore, SerializeUtils)
└── modules/                  # Domain / feature modules
    ├── app.module.ts         # Root AppModule
    ├── api/                  # API feature modules
    │   ├── api.module.ts

    │       ├── controllers/
    │       ├── daos/         # ou repositories/ (Knex)
    │       ├── dto/
    │       ├── entities/

    │       └── <feature>.module.ts

    └── webhooks/             # Webhook handlers

Feature Module Pattern

Location: src/modules/api/<feature>/
Purpose: Cada feature autocontida com seu controller, service, DAO, entidades, DTOs
Exports: Service + DAO para outros módulos consumirem

Core Layer Pattern

Location: src/core/
Purpose: Infraestrutura compartilhada (não colocar lógica de negócio aqui)
Includes: Base classes, decorators, guards, interceptors, error handling

Layer Rules (DDD Anêmico)

Camada O que faz O que NÃO faz
Controller Recebe HTTP, valida DTO, delega ao Service NÃO contém regras de negócio, NÃO acessa DAO direto
Service Toda regra de negócio, orquestração, transformação DTO↔Entity NÃO conhece HTTP (sem @Body, @Param)
DAO/Repository CRUD puro, queries específicas do banco NÃO tem lógica de negócio, NÃO valida
Entity Modelo de domínio com decorators de serialização NÃO tem lógica de persistência, NÃO importa DAO
DTO Shape de input/output da API, validação com class-validator NÃO estende Entity, NÃO tem @JsonObject

Naming Conventions

  • Files (classes/entities): PascalCase — MyEntity.ts, MyFeatureService.ts
  • Files (utilities): kebab-case — base.dao.ts, http-error.ts
  • Components: PascalCase — MyFeatureController, MyEntityDao
  • Methods/Functions: camelCase — create(), findAll()
  • Firestore collections: snake_case, plural — my_entities, deposit_requests
  • PostgreSQL tables: snake_case, plural — subaccount_holders, ledger_entries

Module Wiring

@Module({
  imports: [DependencyModule],
  controllers: [FeatureController],
  providers: [FeatureService, FeatureDao],
  exports: [FeatureService, FeatureDao],
})
export class FeatureModule {}

Regras:

  • Sempre exportar providers que outros módulos precisam
  • Importar módulos, não providers direto
  • Evitar imports circulares — forwardRef() como último recurso
  • DAOs/Repositories ficam na feature, não em core/

TSDTech — API Standards

Philosophy

  • Resource-oriented design com DTOs explícitos
  • Valores financeiros em centavos na API, Decimal internamente
  • Endpoints criptografados por padrão no bus layer
  • Documentação via Swagger (@ApiProperty, @ApiResponse)

Endpoint Pattern

POST /{resource}/{Action}         # Ex: POST /PagamentoPixQrCode/Create

Naming:

  • Ações em PascalCase: Create, Get, Update, List
  • Recurso nomeado como entidade: PagamentoPixQrCode, SubaccountHolder

Request/Response

Request DTO — class-validator + Swagger

export class CreateDepositDto {
  @ApiProperty({ description: 'Valor em centavos' })
  @IsNotEmpty()
  @IsNumber()
  amount: number;  // cents!
}

Success Response

@ApiResponse({ type: MyEntityResponseDto })
async create(@Body() dto: CreateMyEntityDto): Promise<MyEntityResponseDto>

Error Response

{
  "error": {
    "code": "NOT_FOUND",
    "message": "Entity not found",
    "requestId": "trace-id",
    "timestamp": "2026-04-20T10:00:00.000Z"
  }
}

Financial Values (CENTS ↔ DECIMAL)

Regra de ouro:

Momento Formato Biblioteca
Input da API centavos (number) class-validator
Interno (service) Decimal decimal.js
Persistência Decimal @DecimalJsonSerializer()
Output da API centavos (number) service converte 
// Entrada: centavos → Decimal
const amount = new Decimal(dto.amount).div(100);

// Saída: Decimal → centavos
const cents = entity.amount.times(100).toNumber();

Status Codes

  • 201 — Created (POST)
  • 200 — Success (GET, PUT, PATCH)
  • 204 — No Content (DELETE)
  • 400 — Bad Request (validação)
  • 401 — Unauthorized
  • 403 — Forbidden
  • 404 — Not Found
  • 409 — Conflict (estado inválido)
  • 412 — Failed Precondition
  • 500 — Internal Server Error

Encrypted Endpoints

No bus layer (ms-pinbank-bus), cada ação tem duas variantes:

@Post('ActionNameEncrypted')
@PinbankProtected()                      // encrypted: true (default)

@Post('ActionName')
@PinbankProtected({ encrypted: false })  // plain

Ambas chamam o mesmo service method.

Swagger Conventions

  • @ApiTags('FeatureName') no controller
  • @ApiResponse({ type: ResponseDto }) em cada endpoint
  • @ApiProperty({ description: '...' }) em cada campo do DTO
  • Endpoints de health não precisam de auth nem Swagger elaborado

TSDTech — Database Standards

Philosophy

  • DAO/Repository = pure data access, zero business logic
  • Firestore para bus/banking (document store), PostgreSQL para domínio (relacional)
  • AES encryption para dados bancários sensíveis no Firestore

Database per Service

Serviço Database Base Class Collection/Table Pattern
ms-pinbank-bus Firestore BaseDao<T> snake_case, plural: pagamento_pix_qr_code
ms-banking Firestore + AES BaseDao<T> (crypto) snake_case, plural
ms-subaccount PostgreSQL BaseRepository<T, F> (Knex) snake_case, plural: subaccount_holders
ms-report PostgreSQL BaseRepository<T, F> (Knex) snake_case, plural

Firestore DAO Pattern

@Injectable()
export class MyEntityDao extends BaseDao<MyEntity> {
  protected collectionName = 'my_entities';
  constructor(@Inject(Firestore) protected readonly firestore: Firestore) {
    super(firestore, MyEntity);
  }
}

Knex Repository Pattern

interface MyFilter { ids?: string[]; status?: string; }

@Injectable()
export class MyRepository extends BaseRepository<MyEntity, MyFilter> {
  protected tableName = 'my_entities';
  constructor(protected readonly knex: Knex) { super(knex, MyEntity); }

  findBy(query: Knex.QueryBuilder, filters?: MyFilter): void {
    if (filters?.status) query.where('status', filters.status);
  }
}

Naming

  • Collections/Tables: snake_case, plural (subaccount_holders, deposit_requests)
  • FK columns: {table}_id (subaccount_id)
  • Timestamps: created_at_utc, updated_at_utc (timezone-aware)

Entity Decorators

// Firestore
@JsonObject()
export class MyEntity extends MutableEntity {
  @JsonProperty() field: string;
  @DecimalJsonSerializer() amount: Decimal;
}

// PostgreSQL
@JsonObject()
export class MyEntity extends MutableEntity {
  @JsonProperty({ name: 'column_name' }) field: string;
  @KnexDBDate({ name: 'due_date' }) dueDate: Date;
  @KnexDBBool({ name: 'is_active' }) isActive: boolean;
}

Serialization

Database Serializer Methods
Firestore SerializeUtils.serializer serializeObject(), deserializeObject()
PostgreSQL DefaultSerializer serializeObject(), deserializeObjectArray()

Data Integrity

  • NOT NULL, UNIQUE, FK constraints no PostgreSQL
  • IdempotencyKey em entidades de operação (deposit, withdrawal)
  • Ledger entries são imutáveis — nunca atualizar, apenas criar reversões
  • Preferir ImmutableEntity para registros de auditoria

TSDTech — Authentication & Authorization Standards

Philosophy

  • Guards em cadeia (chain of responsibility)
  • Secure by default — múltiplas camadas de auth
  • @PinbankProtected() como decorator unificado

Auth Chain (ordem de execução)

Incoming Request
  → MasterAuthGuard (admin master)
  → AgentAuthGuard (agente/serviço)
  → C2cAuthGuard (client-to-client)
  → GcpAuthGuard (auth interna GCP)
  → MsAuthGuard (microserviço-to-microserviço)
  → @PinbankProtected()

PinbankProtected Decorator

@Post('CreateEncrypted')
@PinbankProtected()                      // encrypted: true (default)
async create(@Body() dto: DTO) { }

@Post('Create')
@PinbankProtected({ encrypted: false })  // plain
async createUnencrypted(@Body() dto: DTO) { }

Endpoint Security Rules

Tipo de Endpoint Auth Exemplo
Health/Readiness Sem auth /health, /health/ready
API pública MasterAuth + MsAuth Endpoints de consulta
API interna (bus) @PinbankProtected() CRUD de operações
Webhooks GcpAuth Callbacks do Pinbank

API-to-API Auth

  • Microserviços se autenticam via MsAuthGuard
  • GCP internals via GcpAuthGuard
  • Tokens com escopo mínimo por serviço

TSDTech — Security Standards

Philosophy

  • Defense in depth: múltiplos guards em cadeia
  • AES encryption para dados bancários em repouso
  • Endpoints criptografados por padrão no bus layer
  • Fail closed — negar por padrão, permitir explicitamente

Encryption

Dados em Repouso (ms-banking)

  • AES encryption para dados bancários sensíveis
  • Transparente via BaseDao<T> com crypto wrapper
  • Decrypt automático na leitura

Dados em Trânsito (Bus Layer)

  • @PinbankProtected() com payload criptografado
  • { encrypted: false } apenas quando explicitamente necessário
  • Ambos os endpoints chamam o mesmo service

Authentication & Authorization

Chain obrigatória de guards:

MasterAuth → AgentAuth → C2cAuth → GcpAuth → MsAuth → @PinbankProtected

Nenhum endpoint (exceto health) deve ignorar a chain.

Secrets & Configuration

  • NUNCA commitar secrets
  • Usar variáveis de ambiente + GCP Secret Manager
  • Validar env vars obrigatórias no startup (fail fast)
  • AES keys gerenciadas externamente (não no código)

Sensitive Data

  • Valores financeiros em centavos na API (nunca expor Decimal raw)
  • Logs: nunca logar tokens, senhas, secrets, PII completo
  • requestId para correlação sem expor dados sensíveis

Security Headers

  • Helmet habilitado em produção
  • Swagger desabilitado em produção
  • CORS configurado por ambiente
  • HSTS em produção

Input Validation

  • class-validator em todos os DTOs
  • Validação global via ValidationPipe do NestJS
  • Sanitização na camada de boundary (controller)

TSDTech — Error Handling Standards

Philosophy

  • Erros de negócio próximos à origem; erros inesperados para handler global
  • Formato consistente em toda a stack
  • HTTPError via métodos estáticos para erros de negócio

Error Shape

{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable message",
    "requestId": "trace-id",
    "timestamp": "ISO-8601"
  }
}

Propagation Pattern

Service → HTTPError

throw HTTPError.NOT_FOUND({ message: 'Entity not found', title: 'Not Found' });
throw HTTPError.FAILED_PRECONDITION({ message: 'Invalid state', title: 'Error' });
throw HTTPError.BAD_REQUEST({ message: 'Invalid input', title: 'Validation Error' });

Global ErrorFilter

  • Captura erros não tratados
  • Retorna 500 + corpo canônico
  • Loga stack trace e requestId

Classification

Categoria HTTP Exemplo
Client (validação) 400 DTO inválido
Auth 401/403 Token ausente/insuficiente
Not Found 404 Entidade não existe
Conflict 409 Estado inválido para ação
Failed Precondition 412 Saldo insuficiente
Server (inesperado) 500 Erro interno

Logging

  • Logar: operation, userId, error code, message, stack, requestId
  • NUNCA logar: passwords, tokens, secrets, full PII
  • Levels: ERROR (falhas), WARN (recuperável), INFO (eventos chave)

Retry

  • Retentar apenas operações idempotentes com erros transientes (5xx, timeout)
  • Nunca retentar 4xx ou erros de negócio
  • Usar idempotencyKey nas entidades

TSDTech — Testing Standards

Philosophy

  • Testar comportamento, não implementação
  • Cobertura crítica em Services (regras de negócio)
  • Testes rápidos e confiáveis

Test Types

Tipo Escopo Mocks Velocidade
Unitário Service isolado DAO/Repository mockado ms
Integração Service + DAO real External APIs mockadas s
E2E Fluxo completo API→DB Nenhum min

Structure (AAA)

it('should reject split when amount exceeds available balance', () => {
  const balance = new Decimal(1000);
  const pendingBalance = new Decimal(-800);

  const result = service.calculateAvailableBalance(balance, pendingBalance);

  expect(result).toEqual(new Decimal(200));
});

Where to Focus

  1. Services — toda regra de negócio, cenários de erro, edge cases financeiros
  2. DTO validation — class-validator cobre validação de input
  3. DAO/Repository — apenas queries customizadas (herdado de base já é testado)
  4. Controllers — mínimos (são thin, só delegam)

Mocking & Data

  • Mockar externals (API Pinbank, Pub/Sub, DAOs)
  • Nunca mockar a classe sob teste
  • Usar factories de entidades com valores padrão
  • Resetar estado entre testes

HTTP Request

Regras de Ouro

Camada Pode NÃO Pode

2. Entity

Base — EntityImmutable / EntityMutable

// src/core/common-models/immutable.entity.ts
@JsonObject()
export abstract class EntityImmutable {
  @JsonProperty()
  private id: string;

  @JsonProperty({
    beforeDeserialize: (p) => new Date(p).toISOString(),
    beforeSerialize: (p) => p?.getTime(),
  })
  private createdAtUtc: Date;

  constructor(params: { id?: string; createdAtUtc?: Date }) {
    if (!params) return;
    this.setId(params?.id);
    this.createdAtUtc = params?.createdAtUtc ?? new Date();
  }

  public setId(id: string) {
    this.id = id ?? crypto.randomUUID();
  }

  public getId(): string { return this.id; }
  public getCreatedAtUtc(): Date { return this.createdAtUtc; }
}

// src/core/common-models/mutable.entity.ts
@JsonObject()
export class EntityMutable extends EntityImmutable {
  @JsonProperty({
    beforeDeserialize: (p) => new Date(p).toISOString(),
    beforeSerialize: (p) => p?.getTime(),
  })
  private updatedAtUtc?: Date;

  constructor(params: { id?: string; createdAtUtc?: Date; updatedAtUtc?: Date }) {
    super(params);
    if (!params) return;
    this.setUpdatedAtUtc(params?.updatedAtUtc);
  }

  public getUpdatedAtUtc(): Date { return this.updatedAtUtc; }
  public setUpdatedAtUtc(d: Date) { this.updatedAtUtc = d; }
  public update() { this.setUpdatedAtUtc(new Date()); }
}

Firestore Entity

// src/modules/api/<feature>/entities/<entidade>.entity.ts
import { JsonObject, JsonProperty } from 'typescript-json-serializer';
import { EntityMutable } from 'src/core/common-models/mutable.entity';
import Decimal from 'decimal.js';
import { DecimalJsonSerializer } from 'src/core/decorators/decimal.decorator';

@JsonObject()
export class MinhaEntidade extends EntityMutable {
  @JsonProperty()
  private nome: string;

  @DecimalJsonSerializer()
  private valor: Decimal;

  @JsonProperty({ name: 'status_atual' })
  private statusAtual: string;

  @JsonProperty()
  private idempotencyKey: string;

  constructor(p: {
    id?: string;
    nome: string;
    valor: Decimal;
    statusAtual: string;
    idempotencyKey: string;
    createdAtUtc?: Date;
    updatedAtUtc?: Date;
  }) {
    super(p);
    if (!p) return;
    this.nome = p.nome;
    this.valor = p.valor;
    this.statusAtual = p.statusAtual;
    this.idempotencyKey = p.idempotencyKey;
  }

  // ── Getters ──
  getNome(): string { return this.nome; }
  getValor(): Decimal { return this.valor; }
  getStatusAtual(): string { return this.statusAtual; }
  getIdempotencyKey(): string { return this.idempotencyKey; }

  // ── Setters ──
  setStatusAtual(s: string) { this.statusAtual = s; }
}

Regras da Entity

  • @JsonObject() na classe + @JsonProperty() em cada campo privado
  • Campos privados com getters públicos (getNome(), getValor())
  • Valores financeiros: @DecimalJsonSerializer() com tipo Decimal
  • Construtor recebe objeto tipado com todos os campos
  • super(p) no início + if (!p) return; (obrigatório para serializacão)
  • Timestamps serializados como Unix timestamp (getTime()) no Firestore
  • NUNCA colocar Swagger, DAO, ou lógica de negócio aqui

BaseDao (v2 — padrão atual)

Métodos do BaseDao

Método Assinatura Descrição
findAll ({ filterIdsAction?, queryFilterAction?, filters?, pagination? }) Busca com filtro duplo (Firestore + in-memory) + paginação cursor
create ({ entities: T[], transaction?: WriteBatch }) Sempre batch — usa batch.set()
update ({ entities: T[], transaction?: WriteBatch }) Sempre batch — usa batch.update()
delete ({ ids: string[], transaction?: WriteBatch }) Sempre batch — usa batch.delete()

Como obter por ID

Não existe findOne(id). Use findAll com filters.ids:

const { items } = await this.dao.findAll({ filters: { ids: [id] } });
const entity = items[0]; // pode ser undefined

Paginação

Cursor-based via lastEvaluatedId:

const result = await this.dao.findAll({
  filters: { ... },
  pagination: new PaginationDto({ lastEvaluatedId: 'abc', limit: 20 }),
});
// result.items → T[]
// result.nextEvaluatedId → string | undefined (para próxima página)

Como implementar um DAO

// src/modules/api/<feature>/daos/<entidade>.dao.ts
import { Injectable } from '@nestjs/common';
import { BaseDao } from 'src/core/base/base.dao';
import { Firestore } from '@google-cloud/firestore';
import { PaginatedListDto } from 'src/core/dtos/paginated-list.dto';
import { PaginationDto } from 'src/core/dtos/pagination.dto';
import { FilterMinhaEntidadeDto } from '../dto/filter-minha-entidade.dto';
import { MinhaEntidade } from '../entities/minha-entidade.entity';

@Injectable()
export class MinhaEntidadeDao extends BaseDao<MinhaEntidade> {
  protected collectionName = 'minhas_entidades';

  constructor(firestore: Firestore) {
    super(firestore, MinhaEntidade);
  }

  async findAll(params: {
    filters?: FilterMinhaEntidadeDto;
    pagination?: PaginationDto;
  }): Promise<PaginatedListDto<MinhaEntidade>> {
    return super.findAll({
      ...params,

      // Filtro in-memory (pós-query) — para campos que não podem ir no where()
      filterIdsAction: (entity) => {
        if (params.filters?.organizationIds &&
            !params.filters.organizationIds.includes(entity.getOrganizationId())) {
          return false;
        }
        return true;
      },

      // Filtro Firestore (pré-query) — usa where() do Firestore
      queryFilterAction: (query) => {
        if (params.filters?.organizationIds) {
          query = query.where('organizationId', 'in', params.filters.organizationIds);
        }
        if (params.filters?.statuses) {
          query = query.where('status', 'in', params.filters.statuses);
        }
        return query;
      },
    });
  }
}

Dual Filter: queryFilterAction vs filterIdsAction

queryFilterAction filterIdsAction
Quando Filters.ids não foi passado Sempre roda (também com ids)
Onde Firestore where() (server-side) In-memory (client-side)
Pra que Filtrar no banco (performance) Pós-filtro para campos não-indexados
Exemplo query.where('status', 'in', statuses) entity.getNome().includes(searchTerm)

Batch + Transaction

Todas as operacões de escrita usam WriteBatch. O transaction é opcional — quando passado, o batch não é committado automaticamente, permitindo compor transacões multi-DAO:

// Auto-commit (caso mais comum)
await this.dao.create({ entities: [entity] });

// Transacão compartilhada entre DAOs
const batch = firestore.batch();
await this.daoA.create({ entities: [a], transaction: batch });
await this.daoB.create({ entities: [b], transaction: batch });
await batch.commit();

// src/modules/api/<feature>/repositories/<entidade>.repository.ts
import { Injectable } from '@nestjs/common';
import { Knex } from 'knex';
import { BaseRepository } from 'src/core/services/base.repository';
import { MinhaEntidade } from '../entities/minha-entidade.entity';

export interface MinhaEntidadeFilter {
  ids?: string[];
  organizationId?: string;
  status?: string;
}

@Injectable()
export class MinhaEntidadeRepository extends BaseRepository<
  MinhaEntidade,
  MinhaEntidadeFilter
> {
  protected tableName = 'minhas_entidades';

  constructor(protected readonly knex: Knex) {
    super(knex, MinhaEntidade);
  }

  findBy(query: Knex.QueryBuilder, filters?: MinhaEntidadeFilter): void {
    if (filters?.ids?.length) {
      query.whereIn('id', filters.ids);
    }
    if (filters?.organizationId) {
      query.where('organization_id', filters.organizationId);
    }
    if (filters?.status) {
      query.where('status', filters.status);
    }
  }
}

// src/modules/api/<feature>/dto/<entidade>-response.dto.ts
import { ApiProperty } from '@nestjs/swagger';

export class MinhaEntidadeResponseDto {
  @ApiProperty() id: string;
  @ApiProperty() nome: string;
  @ApiProperty({ description: 'Valor em centavos', example: 1500 }) valor: number;
  @ApiProperty({ description: 'Data de criacão' }) createdAt: Date;

  static fromEntity(entity: MinhaEntidade): MinhaEntidadeResponseDto {
    const dto = new MinhaEntidadeResponseDto();
    dto.id = entity.getId();
    dto.nome = entity.getNome();

    dto.createdAt = entity.getCreatedAtUtc();
    return dto;
  }
}

7. Service

CREATE

async create(dto: CreateMinhaEntidadeDto): Promise<MinhaEntidadeResponseDto> {
  await this.validateCreate(dto);

  const valorDecimal = new Decimal(dto.valor).div(100);

  const entity = new MinhaEntidade({
    nome: dto.nome,
    valor: valorDecimal,
    statusAtual: dto.status ?? 'ACTIVE',
    idempotencyKey: crypto.randomUUID(),
  });

  const [created] = await this.dao.create({ entities: [entity] });
  return MinhaEntidadeResponseDto.fromEntity(created);
}

FIND BY ID (via findAll com filters.ids)

async findById(id: string): Promise<MinhaEntidadeResponseDto> {
  const { items } = await this.dao.findAll({ filters: { ids: [id] } });
  if (!items.length) {
    throw HTTPError.NOT_FOUND({
      message: 'Entidade não encontrada',
      title: 'Not Found',
    });
  }
  return MinhaEntidadeResponseDto.fromEntity(items[0]);
}

UPDATE

async update(id: string, dto: Partial<CreateMinhaEntidadeDto>): Promise<MinhaEntidadeResponseDto> {
  const { items } = await this.dao.findAll({ filters: { ids: [id] } });
  if (!items.length) {
    throw HTTPError.NOT_FOUND({
      message: 'Entidade não encontrada',
      title: 'Not Found',
    });
  }

  const entity = items[0];
  if (dto.nome !== undefined) entity.setNome(dto.nome);
  if (dto.valor !== undefined) entity.setValor(new Decimal(dto.valor).div(100));

  const [updated] = await this.dao.update({ entities: [entity] });
  return MinhaEntidadeResponseDto.fromEntity(updated);
}

Procedural Mutation (find → modify → update)

async desativar(id: string): Promise<void> {
  const { items: [entity] } = await this.dao.findAll({ filters: { ids: [id] } });
  if (!entity) throw HTTPError.NOT_FOUND({ message: 'Não encontrada', title: 'Not Found' });

  entity.setStatusAtual('INACTIVE');
  await this.dao.update({ entities: [entity] });
}

import { Body, Controller, Delete, Get, Param, Post, Put, Query } from '@nestjs/common';

  2. DAO — estender BaseDao, implementar findAll com filterIdsAction + queryFilterAction

  6. Module — wiring, exports

Checklist de Feature

  • Entity com campos privados + getters públicos
  • Entity usa super(p) + if (!p) return no constructor
  • Valores financeiros com @DecimalJsonSerializer + Decimal
  • DAO com findAll sobrescrito (filterIdsAction + queryFilterAction)
  • DAO sem lógica de négocio
  • Request DTO com class-validator + @ApiProperty (centavos como number)
  • Response DTO com factory fromEntity() (Decimal → centavos)
  • Service usa findAll({ filters: { ids } }) para buscar por ID
  • Service valida com HTTPError estático
  • Create/Update/Delete usam batch (com ou sem transaction)
  • Controller thin (só delega), sem lógica
  • @ApiTags + @ApiResponse em todos endpoints
  • Guard correto (health sem auth)
  • Module com exports dos providers
